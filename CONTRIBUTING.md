# Contributing

The Orchestration Framework is one operator's working methodology, shared publicly so other operators can read it, reference it, and adapt it to their own practice. It is not a community project in the traditional sense — the framework's direction is shaped by lived practice, not by committee.

That said, contributions are welcome within a clear scope.

## What's in scope

- **Typo fixes, broken-link fixes, formatting cleanup.** Open a PR; these get merged quickly.
- **Clarifications where the methodology is ambiguous.** If a passage is genuinely confusing, a PR that sharpens the wording without changing the meaning is welcome.
- **Translations.** If you want to translate a section into another language, open an issue first so we can coordinate placement and naming.
- **Adopter case studies.** If you've adopted the framework and want to contribute a short write-up of how it landed for your context (what worked, what you adapted, what you dropped), open an issue and we'll discuss whether and where it fits.

## What's out of scope

- **Changes to core philosophy, vault structure, agent roles, or rhythms.** These reflect the operator's lived practice. Suggestions are welcome via issue, but a PR that reshapes a foundational document will likely be closed in favor of a discussion.
- **Adding tooling, scripts, or starter kits.** This is a methodology repo, not a template repo. The Template Repository feature is intentionally disabled. Adopters instantiate the framework in their own private vaults; we don't ship runtime tooling here.
- **Vendor-specific integrations.** The framework is deliberately tool-agnostic. PRs that name specific commercial vendors as the recommended tool will be asked to generalize.

## How to contribute

1. **For small fixes** (typos, links, formatting): open a PR directly.
2. **For anything else**: open a GitHub issue first describing what you're proposing. The operator will respond with whether and how to proceed.

## Conduct

Be the kind of contributor you'd want to receive a PR from. Disagreements about methodology are welcome; personal attacks are not.

## Sensitive disclosures

If you find something in this repo that should not be public — accidentally committed personal data, a leaked credential, anything resembling sensitive information — do not open a public issue. Email the operator (contact details on the repo's GitHub profile) so it can be removed before discussion.
